- **change slug**: `33-adaptive-window-ui`

## 1. Why（动机）

LangFix 弹窗当前是**固定尺寸 `NSPanel`（480×460）**，且已支持流式增量渲染（`loading → streaming → result`，见 [../streaming-incremental-render/](../streaming-incremental-render/)）。固定尺寸带来两个体验问题：

1. **尺寸与内容不匹配**：短文本（一行修正）也撑起 460pt 的高窗，大量留白；长文本 + 多条 issue 又被塞进固定高度靠内部滚动。流式场景尤其突兀——内容从零逐字长出，窗口却一开始就是满高。
2. **在场感过强 / 不可暂存**：弹窗浮于所有窗口之上，用户切走去看原文/参考时，要么手动关掉（丢失已生成内容与进行中的流式），要么让它一直挡着。缺少一个「先收起、待会儿再看」的中间态。

期望状态：

- **弹窗高度随流式内容自适应**：初始矮（仅容纳标题栏 + 首行状态），随流式产出**向下增高**，到**屏幕相对高度上限**封顶后内部滚动。宽度固定。
- **失焦即折叠而非关闭**：窗口未关闭前丢失焦点时，**折叠成一个胶囊/小球状的极简入口**（后台流式继续），点击展开恢复；只有显式「关闭」才销毁窗口并取消进行中的请求。
- **科幻 / 艺术感的视觉，且可由用户选主题**：视觉风格做成**多个可选主题**，用户在设置中切换；具体主题的美学定稿交由下游 Codex 设计。

为什么现在做：流式能力（真正的增量渲染）刚落地，正是把「固定尺寸 + 强在场」的旧窗体升级为「自适应 + 可暂存 + 有设计感」的最佳时机；再往后叠加更多内容形态（多语气 refine、术语库）会让固定尺寸的问题更尖锐。

> 本阶段**只产出需求规格（proposal + spec-delta）**，不写代码、不做 UI 设计定稿。UI 设计定稿由下游技术方案阶段（技术方案官 + Codex 交叉评审）完成，用户已明确「以 Codex 最终设计为准」。

## 2. What's Changing（高层变更）

本 change 是弹窗**容器 / 交互 / 视觉外壳**的改造，与 grammar-review 的纠错逻辑正交，因此落到一个**新 capability `review-window`**（现已有架构模块 [review-window](../../architecture/modules/review-window.md)，本 change 首次为其建立 living spec）。

| Capability | 变化类型 | 简述 |
|---|---|---|
| review-window | ADDED | 弹窗尺寸随流式内容自适应：宽高均为屏幕相对范围（宽 `[480, 屏宽×40%]`、高 `[minH, 屏高×70%]`），超上限内部滚动 |
| review-window | ADDED | 三态窗体：展开 / 折叠（胶囊·小球，后台保活）/ 关闭（销毁 + 取消请求）；失焦与 Esc → 折叠 |
| review-window | ADDED | 折叠态用极简 UI（颜色 / 图标）表达「进行中 / 已完成 / 出错」状态 + 极简过渡动画 |
| review-window | ADDED | 视觉主题可选（≥2 套），设置中切换；具体主题美学由下游 Codex 定稿 |
| review-window | ADDED | 取消手动 resize（尺寸完全自动） |

**新增 capability**：`review-window`（判断项，见 §6 决策 D0，若 Reviewer 倾向沿用单 capability，可平移回 grammar-review）。

## 3. Out of Scope（明确不做）

- **不改流式解析 / AI 调用 / 护栏 / diff 等既有逻辑**：只动窗体尺寸、失焦交互、视觉外壳；`ReviewResult` 内容渲染的分区结构不变。
- **不做 UI 视觉定稿**：具体主题的配色 / 材质 / 图标 / 动效曲线由下游 Codex 设计，本阶段只圈定「多主题可选 + 极简动画 + 三态视觉」的边界。
- **不引入自动替换选区 / 历史记录等新功能**（与本次无关，且受 constitution Constraint-2/4 约束）。
- **不做窗口跟随鼠标定位的改造**（出现位置维持现状居中，除非 Codex 设计另有主张，届时另议）。
- **不写代码**：本阶段只产出 proposal + spec-delta。

## 5. Success Metrics（成功指标）

- **尺寸自适应（宽 + 高）**：短内容出小窗、长内容增大到屏幕相对上限后内部滚动；宽高上限均按屏幕相对比例（宽 ≤ 屏宽×40%、高 ≤ 屏高×70%），不因分辨率变化而效果失真。（确定性可测：给定内容与屏幕 visibleFrame，断言窗口宽落在 `[480, visibleW×0.4]`、高落在 `[minH, visibleH×0.7]`，且随内容单调增。）
- **失焦折叠不丢内容**：失焦 / Esc 后窗口折叠为极简入口、流式在后台继续；再展开时内容 = 未折叠时的最新内容。（确定性可测：折叠期间喂入流式增量，展开后断言内容已更新。）
- **关闭即彻底**：点「关闭」销毁窗口**并取消进行中的 AI 请求**（不再有后台请求继续跑）。（确定性可测：关闭后断言底层 Task 被 cancel。）
- **状态可辨识**：折叠态下「进行中 / 已完成 / 出错」三态有可区分的颜色 / 图标。（确定性可测：断言三态映射到不同视觉标识。）
- **主题可选**：设置中提供 ≥2 套视觉主题、可切换、切换即时生效、有默认主题。（确定性可测：切换主题后断言当前主题标识变化。）

## 6. Clarifications（Clarify 阶段——用户已逐条拍板）

用户在 RAS-34 评论中对 5 个决策点逐条给出结论，摘录并转成约束如下。

### D0（用户已确认）: capability 归属
弹窗外壳与 grammar-review 纠错逻辑正交，本 change 落到**新 capability `review-window`**。此前作为需求官判断项抛给用户，用户回复「其他的我没有问题了」**接受此归属**。（若编排官仍倾向沿用单 capability，可无损平移改目录名。）

### Q1: 尺寸规则（宽度 / 高度 / 上限）——含用户二次补充
**用户拍板（两轮）**:
- 第一轮：接受「宽度固定、高度动态」，**上限必须用屏幕相对高度**，不用固定 pixel。
- 第二轮补充：**宽度也改为屏幕相对范围**——「最低宽度不会小于 480，最大不超过屏幕的 40%」。用户同时确认接受需求官建议的高度比例 0.7。

**转成约束**:
- 宽度 = 内容自然宽度，clamp 到 `[minW, maxW]`；`minW = 480pt`，`maxW = 所在屏 visibleFrame.width × 0.4`。
- 高度 = 内容自然高度，clamp 到 `[minH, maxH]`；`minH` 为容纳标题栏 + 首行状态的内容自然高度（内容驱动，天然分辨率无关）；`maxH = 所在屏 visibleFrame.height × 0.7`。
- 宽或高超过各自上限后，内容区内部滚动。
- 尺寸变化过程走极简动画（与 Q2 一致）。

> **`minW` 取值说明（需求官判读）**: 用户原话为「最低宽度不会小于 48」，需求官判读为 **480pt**——理由：现状固定宽度即 480，48pt 的展开态窗口无法容纳一行修正文本，属明显笔误漏一个 0。已在回报中显式标注此判读，若用户确为 48 会一句话纠正（改动极小）。

### Q2: 折叠入口形态与状态提示
**用户拍板**: 折叠成**胶囊或小球**；用**简洁 UI（颜色 / 图片）表达加载是否完成**——进行中一种颜色/图标，完成另一种；**动画用极简动画**。

**转成约束**:
- 折叠态 = 缩窄的胶囊 / 小球（**不保持展开态宽度**），具体形态由 Codex 设计。
- 折叠态至少可视化区分三态：**进行中（流式/loading）/ 已完成（result）/ 出错（error）**，用颜色 + 图标区分（需求官补充 error 态，因 `ReviewState.Phase` 含 `.error`）。
- 折叠 ↔ 展开、状态切换均为**极简动画**（无炫技长动效）。

### Q3: 折叠 vs 关闭 语义
**用户拍板**: **失焦 → 折叠**（后台继续）；**关闭 → 关闭所有窗口并取消当前请求**；**Esc 默认等同于失焦（即折叠，不是关闭）**。

**转成约束**（三态机）:
| 触发 | 结果 |
|---|---|
| 失焦（窗口 resignKey） | 折叠为入口，**不销毁**，流式/请求后台继续 |
| Esc | 等同失焦 → 折叠 |
| 点击折叠入口 | 展开恢复 |
| 关闭按钮 / 取消 | 销毁窗口 + **取消进行中的 AI 请求**（cancel 底层 Task） |

> **与现状差异（需求官核对，纠正派单描述）**: 派单称「现状 resignKey 自动关闭」，但 master（流式 feat/16 已合入）弹窗 `hidesOnDeactivate=false` 且**无任何失焦处理**——当前失焦不关闭，Esc 才关闭。故本 change 是：**新增失焦→折叠**，并把 **Esc 从「关闭」改为「折叠」**，把「关闭」语义强化为「销毁 + 取消请求」。

### Q4: 「科幻 / 艺术感」视觉
**用户拍板**: 路线 A（静态材质）与路线 B（A + 克制动效）**都不错，做成可选主题让用户选**；希望 **Codex 多设计几套**，**以 Codex 最终设计为准**（用户自陈无美学基础）。

**转成约束**:
- 需求层只约束：**提供 ≥2 套视觉主题**、设置中可切换、有默认主题、切换即时生效。
- 主题应体现「科幻 / 艺术感」调性（毛玻璃 / 发光描边 / 暗色科技配色 / 几何字体等为候选素材），**具体主题集合与美学细节由下游 Codex 设计定稿**，本阶段不锁死。
- 明暗主题兼容性 = 设计阶段决定项（需求层不强制，但建议默认主题在明/暗环境均可读）。

### Q5: 流式自动增高 vs 手动 resize
**用户拍板**: 采用流式自动增高，**关闭手动 resize**。

**转成约束**: 移除窗体 `.resizable`，尺寸完全由内容/流式自动决定，无手动拉伸。

## 7. 风险

| 风险 | 可能性 | 影响 | 缓解 |
|---|---|---|---|
| 高度随流式逐帧变化导致「窗口抖动 / 频繁重排」 | 中 | 中 | 需求层约束「极简动画平滑过渡」；增高节流 / 动画曲线归设计阶段 |
| 折叠态后台流式继续，但用户以为已停止（心智不一致） | 中 | 中 | Q2 三态视觉：折叠态用颜色/图标明确「进行中 vs 已完成」，完成时给可辨识提示 |
| 「关闭取消请求」与既有 `onCancel/onClose` 回调语义重叠或遗漏取消 | 中 | 中（正确性：请求泄漏后台跑） | spec 明确「关闭 = 销毁 + cancel 底层 Task」；设计阶段核对现有 `onCancel/onClose/currentTask` 汇聚到单一取消路径 |
| 多主题切换引入状态/持久化复杂度（主题存哪、默认值） | 低 | 低 | 主题选择属非敏感偏好，写 UserDefaults（不进 Keychain，遵 Constraint-1）；默认主题 = Codex 指定 |
| 屏幕相对上限在多屏/外接显示器/分辨率切换时取错屏 | 低 | 中 | spec 约束「以弹窗所在屏的 visibleFrame 为基准」；多屏细节归设计阶段 |
| 失焦→折叠 与 macOS 焦点抖动（临时子面板/输入法窗）误触发折叠 | 低 | 中 | 设计阶段甄别真实失焦（非本 App 附属窗口）；需求层标注该边界 |
