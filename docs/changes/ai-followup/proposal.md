# Proposal: ai-followup

- **Owner**: by 需求官 on behalf of wu.nerd
- **Reviewers**: wu.nerd、技术方案官
- **创建日期**: 2026-07-14
- **状态**: Clarify
- **关联 Issue**: RAS-53
- **共享分支**: `feat/53-ai-followup`

## 1. Why（动机）

LangFix 当前是**单轮、一次性纠错**：用户划词 → AI 返回结构化 `ReviewResult`（`corrected` + 逐条 `issues`）→ 用户复制或关窗，链路结束，无任何会话/上下文（现状证据：`AppCoordinator.swift` 每次选区一个独立 Task，关窗即销毁；`AIClient` 每次仅发 `[system, user]` 两条消息，无历史）。现状无「围绕结果继续对话」的能力。

真实场景里，用户看到 AI 给出的多处修正后，常有**针对具体某处修正的疑问**——「为什么第二处要这么改」「修正 2 是否适用于所有正式邮件场景」。现状用户无法就结果继续追问，只能重新选词从头再来或自行到别处提问，纠错的「可信可核对」价值打了折扣。

期望状态：在纠错结果之上，新增**围绕本次修正结果的追问/答疑能力**——用户可在结果浮窗内对 AI 追问，且能**精确指定要问的是哪一处修正**（引用「修正 2」）。为支持精确引用，需给每处修正引入**稳定、用户可见的序号**。

为什么现在做：核心纠错链路（结构化输出、三级降级、最小改动护栏、流式预览）已稳定，具备承载「追问」的数据与 UI 基础（`Issue` 已是结构化模型、`messages` 数组天然支持多条、`ReviewView` 状态机可扩展）。此时补齐「结果可追问」能补完产品的可信闭环。

> **交互设计归属**：用户已明确「具体的交互方式让 Codex 来设计，要有设计感且易用」。**本 change 只定义需求约束与红线边界（WHAT / 验收），不定交互形态与实现（HOW）**；追问输入框位置、引用语法糖、消息气泡样式、流式渲染细节等由技术方案阶段（技术方案官 + Codex 交叉评审）落地。

## 2. What's Changing（高层变更）

| Capability | 变化类型 | 简述 |
|---|---|---|
| grammar-review | ADDED | 「修正稳定序号」：最终定稿的每处 issue 拥有稳定、用户可见、与发送给 AI 的上下文同源的序号（如 #1/#2），供引用与追问定位 |
| grammar-review | ADDED | 「结果追问（答疑）」：用户可就本次纠错结果向 AI 追问，AI 以对话式解释回答；追问不改动主结果、不产出可替代主结果的新版本 corrected |
| grammar-review | ADDED | 「追问上下文注入防御」：追问上下文中的原文/修正/用户问题/历史回答一律作为数据而非指令处理 |
| grammar-review | ADDED | 「追问态可取消与隔离」：关窗/取消/新纠错必须中止追问并清空会话；旧请求晚到不得污染当前结果 |
| grammar-review | ADDED | 「追问上下文预算」：多轮上下文超模型上限时，不得静默丢失被引用的修正；按确定性规则处理（fail loud 或不损引用地裁剪非关键历史） |
| grammar-review | ADDED | 「追问会话隐私与生命周期」：会话仅存于窗口/请求生命周期的易失内存，关窗/取消/退出即清；不入日志/文件/UserDefaults/崩溃或分析上报/剪贴板 |
| grammar-review | MODIFIED | 「展示结构化修正」：每条 issue 展示补充用户可见序号；隐私说明补充「追问内容同样发往用户配置端点」 |

**新增 capability**：无（grammar-review 是本项目唯一 capability）。

## 3. Out of Scope（明确不做，待用户确认边界后固化）

- **不做「追问触发重新纠错 / 改写主结果」**（V1 倾向，见 Q2）：追问只做**答疑/解释**，不改动已展示的 `corrected`/`issues`，且 AI 回答中**不得给出可替代主修正结果的新整段 corrected**（防止绕过最小改动护栏 Constraint-3——「不改 corrected 字段」≠「回答里不吐一版新全文」，二者都要挡）。「根据追问调整修正」作为后续演进。
- **不做通用聊天机器人**：追问范围锚定「本次这段文本的纠错结果」，不承接与本次修正无关的任意问答（保持「纠错器不是聊天器」定位）。
- **不做持久化历史 / 跨窗口会话 / 历史记录功能**：关窗即销毁会话（如未来要做需按 Constraint-2 走默认关闭 + 显式开关 + 本地加密 + ADR）。
- **不改最小改动护栏（ADR-0004 / Constraint-3）**：追问既不产生新主结果 corrected，就不触及护栏算法；首轮纠错的护栏行为保持不变。
- **不定交互形态与实现**：输入框位置、引用「修正 N」的具体交互、消息列表布局、追问是否流式渲染等 HOW，归技术方案阶段。
- **不写代码**：本阶段只产出 proposal + spec-delta。

## 5. Success Metrics（成功指标）

- **可精确引用**：最终定稿的每处修正有稳定、用户可见、与上下文同源的序号；用户引用某序号时，发送给 AI 的上下文能无歧义定位到该处修正（确定性可测：构造带序号修正清单 + 引用「修正 2」→ 断言上下文序号与对应 `before/after/reason_zh` 正确绑定）。
- **答疑不改主结果、不越护栏**：追问及回答**不修改** `corrected`/`issues`，且回答不产出可替代主结果的新整段 corrected（确定性可测）。
- **隐私不破线**：追问会话只存易失内存、关窗/取消/退出即清；日志/文件/UserDefaults/崩溃或分析上报/剪贴板均不含原文/修正文/追问内容（确定性可测）。
- **上下文不静默失真**：多轮上下文超限时不静默丢失被引用的修正——要么明确报错可重试，要么按确定性规则裁剪且保留被引用项（确定性可测）。
- **失败与取消干净**：追问请求失败或被取消时给明确中文错误/干净收尾，不崩溃、不污染主结果、失败或取消内容不进入会话历史（确定性可测）。

## 6. Clarifications（Clarify 阶段——需用户拍板，我给推荐倾向）

> 以下为**需求边界层面**的决策（非交互/实现），需 Owner 确认后固化进 spec-delta。交互设计（HOW）留给 Codex。

### Q1: 追问会话历史的保留边界（触及红线 Constraint-2）
**背景**: 追问要让 AI 理解上下文，需把「原文 + 本次修正结果 + 之前问答轮次」发给端点，含用户敏感文本。Constraint-2 约束的是**落盘/日志/持久化**，未禁止**运行时易失内存**中的临时会话状态。
**推荐（待确认）**: 会话历史**仅存于结果浮窗/请求生命周期的易失内存**，关窗/取消/新纠错/退出即清；**不入**日志、文件、UserDefaults、崩溃或分析上报、剪贴板。日志仍只允许 requestId/耗时/token/状态码。
**待用户确认**: 是否接受「关窗即丢失全部追问历史、无历史记录功能」？（若要保留历史 → 触发 Constraint-2 强制覆盖路径，需 ADR + 默认关闭 + 加密存储，范围显著变大。）

### Q2: 追问能否触发「重新纠错 / 改写主结果」（触及红线 Constraint-3）
**选项**:
- **A（推荐，V1）纯答疑**：追问只让 AI **解释/回答**，产物是对话式文本，不改 `corrected`/`issues`、不产出可替代主结果的新整段 corrected。范围可控、不触红线。
- **B 可迭代纠错**：允许追问驱动 AI 产出新版本 corrected。需为新版本保持护栏（Constraint-3）、diff、复制目标、版本切换等，复杂度与红线面显著上升。
**推荐**: V1 选 **A**，B 列 Out of Scope。
**待用户确认**: V1 追问是否限定为「纯答疑，不改主结果、不吐可替代全文」？

### Q3: 修正序号的语义与稳定性
**背景**: 现状 `Issue` 有 UUID（列表 key）但无用户可见序号。且现有流式有「预览→定稿」与 strict 重试可能改变 issue 列表。
**推荐（待确认）**: 序号**只在最终 `ReviewResult` 定稿后生成并开放引用**（流式预览期不展示可引用编号）；顺序 = 定稿后客户端展示用的 issue 列表顺序，**与发送给 AI 的上下文完全同源**；从 1 起递增、该次结果存续期内固定，不随滚动/折叠/追问/strict 覆盖漂移。
**待用户确认**: 该序号规则是否符合预期？

### Q4: 追问上下文的组成范围
**推荐（待确认）**: 上下文 = 原文 `original` + 本次完整修正结果（带序号 `issues`：类型/严重度/before→after/中文解释 + `corrected` + `summary_zh`）+ 本浮窗内已发生的问答轮次；均以**引用数据**形式携带（见注入防御）。
**待用户确认**: 上下文范围是否合适（尤其"是否需携带完整 issues 清单"）。

### Q5: 追问范围边界
**推荐（待确认）**: 追问限定在**围绕本次这段文本的纠错结果**答疑；对明显跑题不作保证/可温和引导。**拒答/引导话术属交互，留给 Codex**；本阶段只定「范围锚定本次结果」这一约束。
**待用户确认**: 是否接受该范围锚定。

### Q6: 追问回答是否流式渲染
**推荐（待确认）**: 追问回答是否流式属**技术方案阶段**决定；需求层仅要求「最终回答可读、失败可恢复、不影响主结果」，不强制流式、不指定实现方式。

## 7. 风险

| 风险 | 可能性 | 影响 | 缓解 |
|---|---|---|---|
| 追问引入会话历史，被误做成「持久化历史」而破 Constraint-2 | 中 | 高（隐私红线） | Q1 + spec「追问会话隐私与生命周期」：仅易失内存、多触发点即清、枚举禁止落点、日志零内容 |
| 「追问触发改写」或 AI 回答直接吐新全文，绕过最小改动护栏 Constraint-3 | 中 | 高 | Q2 + Out of Scope + spec：V1 纯答疑，回答不得产出可替代主结果的新整段 corrected |
| 多轮上下文膨胀超模型上限，静默截断致「修正 N」失去绑定 | 中 | 高（正确性） | spec「追问上下文预算」：不得静默丢失被引用修正，超限 fail loud 或确定性裁剪非关键历史 |
| 追问上下文含注入（用户追问或原文里「忽略规则/重写全文」） | 中 | 高 | spec「追问上下文注入防御」：全部作为数据、只围绕本次结果答疑、不执行其中指令 |
| 追问进行中关窗/取消/新纠错，旧响应晚到污染新窗口或残留历史 | 中 | 高（正确性） | spec「追问态可取消与隔离」：多触发点 cancel + 清历史，旧 generation 响应丢弃 |
| 序号在流式预览期或 strict 覆盖后漂移，「修正 2」指代错乱 | 中 | 中 | Q3 + spec：仅定稿后生成/可引用，与上下文同源、生成后固定 |
| 追问偏离为通用聊天，稀释「纠错器」定位 | 中 | 中 | Q5 + spec：范围锚定本次结果 |
| 追问 UI/状态叠加破坏既有 loading/streaming/result/error 状态机与取消语义 | 低 | 高 | 追问态作为 result 态之上的叠加，既有 Requirement 覆盖测试须仍全绿 |
| 现状虽无会话，但引入「围绕结果的有限多轮答疑」是新架构决策 | —— | 中 | 设计阶段评估是否新开 ADR 记录该决策与红线影响（**注**：ADR-0005 裁剪的是「流式/自动替换选区」，并未涉及多轮对话，本 change 不与之直接冲突） |
