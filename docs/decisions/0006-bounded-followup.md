# ADR-0006: 引入「围绕结果的有限多轮追问答疑」

- **状态**: Accepted
- **日期**: 2026-07-14
- **Owner（决策者）**: n374
- **Reviewers**: n374（自审）、技术方案官 + Codex（对抗式评审）
- **关联 change**: [ai-followup](../changes/ai-followup/)
- **影响 capability**: grammar-review

## 1. 上下文

LangFix V1 是**单轮、一次性纠错**：划词 → AI 返回结构化 `ReviewResult` → 用户复制/关窗，链路结束，**无任何会话/上下文**（`AppCoordinator` 每次选区一个独立 Task 关窗即销毁；`AIClient` 每次仅发 `[system, user]` 两条消息、无历史）。ADR-0005 明确 V1 砍掉「流式」「自动替换选区」，但**未涉及多轮对话**。

RAS-53 需求：用户看到多处修正后常有针对某处的疑问（「修正 2 是否适用于所有正式邮件场景」），现状无法就结果继续追问。为补齐「结果可信可核对」的闭环，需在纠错结果之上**新增围绕本次结果的追问答疑**，并给每处修正**稳定可见序号**以便精确引用。

这使 LangFix 从**单轮 → 有限多轮**，是一次架构级方向变化，需 ADR 留痕（proposal §7 已预告设计阶段评估是否新开 ADR）。

## 2. 决策

引入**有限多轮追问答疑**能力，严格锚定以下**边界**（本决策 = 能力新增 + 边界约束，二者不可分）：

1. **纯答疑，不改主结果**：追问只让 AI 以对话式文本解释；**不修改**已展示的 `corrected`/`issues`，**不产出可替代主结果的新整段 corrected**（守 Constraint-3；「追问驱动重纠错」列 Out of Scope）。
2. **话题范围：语法/语言相关**（**2026-07-17 修订**，原「范围锚定本次结果」，见 language-config change / RAS-57）：放宽为可解答**任意语法 / 用词 / 语言相关问题**（不限于本次结果直接相关）；与语言无关的通用闲聊仍不在职责内，不做通用聊天机器人。放宽仅及话题范围——边界 1/3/4/5/6 全部原样保留，且不得因话题放宽输出可替代主结果的整段 corrected。修订前原文：「只保证『围绕本次这段文本的纠错结果』的答疑，不做通用聊天机器人。」
3. **易失内存、关窗即清**：追问会话历史仅存于结果浮窗/请求生命周期的易失内存；关窗/取消/新纠错/退出即清；**不落盘、不入日志/UserDefaults/文件/剪贴板/崩溃或分析上报**（守 Constraint-2）。
4. **稳定同源序号**：每处修正的用户可见序号 = 定稿 `issues` 数组下标，UI 显示与发给 AI 的上下文编号**同源**，定稿后不漂移。
5. **注入防御延伸**：追问上下文（原文/修正/历史/当前问题）一律作为**数据**处理，不执行其中指令（延伸 living spec「输入视为数据」到追问链路）。
6. **上下文预算正确性**：多轮上下文超模型上限时，**不得静默丢失被引用的修正**——保留 base（原文+定稿结果+被引用修正+当前问题）、仅裁剪非关键历史，放不下则 fail loud。

追问回答**流式产出 + 尽力 Markdown 渲染**（用户明确要求）。

## 3. 理由

- **补可信闭环**：纠错的价值在「可信、可核对」；能就某处修正追问「为什么这么改」，直接强化该价值，成本集中在一个新的「结果附属层」，不动首轮纠错链路。
- **红线只增不减**：本决策**不弱化任何现有红线**——反而把 Constraint-2/3 与注入防御**延伸**到新链路，属宪法「红线只增不减」下的正常能力演进。
- **与 ADR-0005 不冲突**：0005 裁剪的是「流式输出」「自动替换选区」，**未**约束「是否多轮」。本 change 的流式仅用于**追问回答呈现**（非首轮结构化流式，那由 streaming-incremental-render change 承载），且不触碰「替换选区」（Constraint-4 保持）。
- **有限而非开放**：「围绕本次结果、纯答疑、易失内存」三重边界把「多轮」限制在可控范围，避免滑向通用聊天与持久化历史（后者才是 Constraint-2 的强制覆盖路径）。

## 4. 后果

- **正面**：结果可追问、可精确引用，补齐可信闭环；追问为 result 态之上的纯叠加，可独立回滚，不影响首轮纠错。
- **负面**：新增会话状态、取消/代次隔离、上下文预算、输出护栏等复杂度；追问内容同样经 HTTPS 外发到用户配置端点（须在隐私说明明示，扩展 NFR-5）。
- **中立**：「追问驱动重纠错 / 持久化历史」列为后续演进，若要做需各自新开 ADR（重纠错评估 Constraint-3；持久化评估 Constraint-2 强制覆盖路径）。

## 5. 备选方案

| 方案 | 优点 | 缺点 | 为什么不选 |
|---|---|---|---|
| 不做追问，维持单轮 | 最简单、零新增红线面 | 用户对具体修正的疑问无处安放，可信闭环缺口 | 放弃核心价值增益 |
| 追问可驱动重纠错/改写主结果 | 更强大 | 需为新版本保持护栏、diff、版本切换，红线面显著上升 | V1 复杂度与 Constraint-3 风险过大，列后续 |
| 追问历史持久化（历史记录功能） | 可回溯 | 触发 Constraint-2 强制覆盖路径（默认关闭+加密+ADR） | 范围显著变大，V1 不做 |
| **有限多轮、纯答疑、易失内存（本决策）** | 补闭环且边界可控、不弱化红线 | 仍有会话/取消/预算复杂度 | 收益与风险最平衡 |
